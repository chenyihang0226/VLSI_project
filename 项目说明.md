# VLSI Lab1 项目说明

本文档根据当前工作区代码重新整理，面向刚接触该项目的同学，说明项目结构、运行流程，以及 `solution.cpp` 中当前多层次超图划分算法的实现原理。

## 1. 项目目标

本项目实现的是 VLSI 网表的**超图二分划分**。

给定一个超图：

- `Node` 表示顶点，可以理解为电路单元或模块。
- `Net` 表示超边，可以连接多个顶点，可以理解为一条线网。
- 程序要把所有顶点分成两个集合 `X` 和 `Y`。
- 两个集合需要满足 `50% ± 2%` 的平衡约束。
- 优化目标是让被切开的超边数量尽量少。

如果某条 `Net` 同时连接了 `X` 中的点和 `Y` 中的点，这条 `Net` 就被切开。所有被切开的 `Net` 数量就是 `cutsize`，也叫割代价。

## 3. 输入与输出格式

当前程序只使用 ICCAD 2021 TopoPart 格式的 benchmark 文件（位于 `ICCAD2021-TopoPart-Benchmarks/Generated Benchmarks/` 目录下）。

### 3.1 输入格式

以 `case1` 为例，文件结构如下：

```text
300000          <-- 第 1 行：顶点数量
749091          <-- 第 2 行：超边数量
212496 250542   <-- 第 3 行起：每条超边连接的顶点编号（0 索引）
85805 88841
4047 221825
...
```

格式说明：

1. **第 1 行**：顶点数量 `n1`
2. **第 2 行**：超边数量 `n2`
3. **第 3 ~ 3+n2-1 行**：每行表示一条超边，包含该超边连接的所有顶点编号，以空格分隔。**顶点编号从 0 开始**（0-based indexing）。
4. **第 3+n2 行 ~ 末尾**：固定节点信息（每个 FPGA 上固定的节点集合）。当前二分划分算法不使用此部分信息。

### 3.2 输出 `results/caseX_partition.txt`

输出到 `lab1/results/` 目录，文件名与输入基准名对应。每一行对应一个顶点：

```text
第 1 行 -> node 1
第 2 行 -> node 2
...
第 i 行 -> node i
```

每行内容：

```text
0 表示该点属于 X
1 表示该点属于 Y
```

## 6. benchmark 读取逻辑

位置：`Solution::read_benchmark`。

当前代码只支持 **ICCAD 2021 TopoPart 格式**，与旧 `.hgr` 格式的不同点：

| 项目 | 旧 `.hgr` 格式 | ICCAD 2021 格式 |
|------|---------------|-----------------|
| 第 1 行 | `线网数 顶点数`（两个数字） | `顶点数`（一个数字） |
| 第 2 行 | 第一条线网 | `线网数`（一个数字） |
| 第 3 行起 | 后续线网 | 第一条线网 |
| 顶点编号 | 1 索引（1-based） | **0 索引（0-based）** |
| 文件末尾 | - | 固定节点信息（用于 FPGA，本算法忽略） |

读取逻辑：

```cpp
// 第 1 行：顶点数量
getline(file >> ws, line);
iss >> node_num;

// 第 2 行：线网数量
getline(file >> ws, line);
iss2 >> edge_num;

// 后续 edge_num 行：每条线网
for (int i = 0; i < edge_num; i++) {
    getline(file, line);
    istringstream iss(line);
    int node_id;

    Net *net = graph.add_net(i);

    while (iss >> node_id) {
        node_id++;  // 0 索引 -> 1 索引转换
        Node *node = graph.get_or_create_node(node_id);
        node->add_net(net);
        net->add_node(node);
    }
}
```

注意：
1. 读取线网时，文件中 0 索引的节点编号通过 `node_id++` 转为 1 索引，与 `Graph` 内部存储一致。
2. 文件末尾的固定节点信息不被读取，因为二分划分算法不需要此信息。

## 7. 当前 solution.cpp 的总体算法

当前 `solution.cpp` 已经重构成了一个**多层次超图划分 + FM 细化**算法。

一句话流程：

```text
Graph 指针式超图
  -> 转成内部紧凑 Hypergraph
  -> 多次重启
  -> 逐层粗化
  -> 最粗层做多种初始划分
  -> FM 细化
  -> 逐层反粗化并继续 FM
  -> 保留 cutsize 最小的结果
  -> 转回 X/Y 集合
```

主要参数：

| 参数                        |   当前值 | 含义                         |
| --------------------------- | -------: | ---------------------------- |
| `kBalanceRatio`           | `0.02` | 平衡约束，即 `50% ± 2%`。 |
| `kRestarts`               |    `8` | 外层随机重启次数。           |
| `kMaxLevels`              |    `5` | 最多粗化层数。               |
| `kMinCoarsestNodes`       |  `320` | 粗图小于该规模后停止粗化。   |
| `kStopCoarsenRatio`       | `0.92` | 粗化缩减不明显时停止。       |
| `kBudgetBaseTime`        |   `90` | 自适应预算基准（秒），对应 ~12k 节点 |
| `kBudgetBaseNodes`       | `12000` | 基准节点数                     |
| `kBudgetExponent`        |  `0.8` | 预算规模缩放指数（<1 次线性） |
| `kBudgetMinSeconds`      |   `60` | 最小时间预算（秒）             |
| `kBudgetMaxSeconds`      | `3600` | 最大时间预算（秒，1 小时封顶） |
| `kHeavyNetLimit`          |   `64` | 粗化匹配时忽略过大的 net。   |
| `kSmallNetBoost`          |  `1.7` | 小 net 匹配加权。            |
| `kMatchDegreePenalty`     | `0.06` | 匹配高连接度点时的惩罚系数。 |
| `kInitModeCount`          |    `3` | 最粗层三种初始化策略。       |

## 8. 内部 Hypergraph 结构

虽然助教框架用 `Graph/Node/Net` 指针结构，算法内部又构造了一个更适合高速计算的 `Hypergraph`：

```cpp
struct Hypergraph {
    int n = 0;
    vector<vector<int>> nets;
    vector<vector<int>> node_nets;
    vector<int> node_weight;
};
```

含义：

- `n`：当前层的节点数。
- `nets[i]`：第 `i` 条 net 连接的所有节点，使用内部下标。
- `node_nets[u]`：节点 `u` 关联的所有 net。
- `node_weight[u]`：节点权重。原始图中每个点权重为 `1`，粗化后权重会累加。

为什么要转成这个结构？

- `Graph` 中保存的是指针，适合表达对象关系，但计算时访问开销较大。
- `Hypergraph` 用连续数组和整数下标，适合频繁遍历和增量更新。
- FM 细化时需要大量查询“一个点连接哪些 net”“一个 net 连接哪些点”，双向索引很方便。

`build_fine_hypergraph` 会完成：

1. 收集所有原始 node id。
2. 排序，建立 `原始 id -> 内部下标`。
3. 把每条 `Net` 转换成内部下标形式。
4. 对 net 内节点去重。
5. 过滤只含一个节点的无效 net。
6. 调用 `rebuild_node_nets` 建立 `node_nets`。

最后还会保存 `idx_to_id`，用于算法结束后把内部下标转回原始 node id。

## 9. 粗化 coarsen_once

粗化的目的是把强相关的节点合并成“超级节点”，让问题规模变小。

当前位置：`coarsen_once`。

大致步骤：

1. 所有节点一开始都是 unmatched。
2. 随机打乱节点顺序。
3. 再按节点连接的 net 数量降序稳定排序，让高度数节点优先处理。
4. 对每个未匹配节点 `u`，从共享 net 的邻居里找一个最适合匹配的 `v`。
5. 找到就把 `{u, v}` 合成一个粗节点。
6. 找不到就让 `{u}` 单独成为一个粗节点。

匹配评分核心：

```text
score(u, v) += net_weight / degree_penalty
```

其中：

```text
net_weight = 1 / (net_size - 1)
```

含义是：越小的 net 越重要，因为小 net 往往代表更紧密的连接关系。

如果 `net_size <= 4`，代码还会乘上：

```cpp
kSmallNetBoost = 1.7
```

也就是进一步保护小 net。

度数惩罚：

```text
degree_penalty = 1 + 0.06 * degree(v)
```

这样高度数节点不会过于容易被合并，避免粗化阶段形成过大的超级节点。

粗化后还要重建 net：

1. 原 net 中每个细节点映射到粗节点。
2. 排序去重。
3. 如果一条 net 映射后只剩一个粗节点，说明它已经完全在同一个粗节点内部，不再参与 cut 计算，因此过滤掉。

## 10. 三种初始划分

最粗层规模较小，代码会尝试三种初始划分方式。

### 10.1 度数平衡初始化

函数：`init_degree_balanced_part`

做法：

1. 随机打乱节点。
2. 按节点度数降序排序。
3. 优先处理连接更多 net 的节点。
4. 在满足平衡约束的前提下，把节点放到当前更轻的一侧。

直觉：高度数节点影响更多 net，应该更早决策。

### 10.2 随机平衡初始化

函数：`init_random_balanced_part`

做法：

1. 随机打乱节点。
2. 在满足平衡约束的前提下，尽量把节点放到较轻的一侧。

直觉：提供解的多样性，避免每次都从同一个局部区域开始搜索。

### 10.3 小网权重初始化

函数：`init_smallnet_balanced_part`

做法：

1. 先给每个节点计算一个小网相关的 `score`。
2. 节点所在 net 越小，对 score 贡献越大。
3. 按 score 从高到低处理节点。
4. 同时平衡两边累计权重和累计 score。

直觉：小 net 更容易体现强连接，把这类节点处理好有助于降低 cutsize。

## 11. cut 与 delta 的计算

### 11.1 每条 net 的两侧计数

函数：`build_net_side_counts`

它会统计：

```text

count_x[i] = 第 i 条 net 中有多少节点在 X
count_y[i] = 第 i 条 net 中有多少节点在 Y
```

一条 net 被切开的条件是：

```text
count_x[i] > 0 && count_y[i] > 0
```

`cut_from_counts` 就是遍历所有 net，统计满足这个条件的 net 数量。

### 11.2 delta_cut_for_move

函数：`delta_cut_for_move`

它预测：如果把节点 `u` 移到另一边，cutsize 会变化多少。

返回值含义：

```text
delta < 0  移动后 cutsize 下降
delta = 0  移动后 cutsize 不变
delta > 0  移动后 cutsize 上升
```

它只遍历 `u` 连接的 net：

```cpp
for (int net_idx : hg.node_nets[u]) {
    int cx = count_x[net_idx];
    int cy = count_y[net_idx];
    bool before_cut = (cx > 0 && cy > 0);

    if (from_x) {
        cx--;
        cy++;
    } else {
        cy--;
        cx++;
    }

    bool after_cut = (cx > 0 && cy > 0);
    delta += after_cut - before_cut;
}
```

因为移动 `u` 只会影响它连接的 net，其他 net 的 cut 状态不可能变化。

在 FM 里，代码使用：

```cpp
gain[u] = -delta_cut_for_move(...)
```

所以：

```text
gain 越大，说明移动这个点越有利。
```

## 12. FM 细化 fm_refine_sampled

FM 细化是当前算法最核心的局部优化模块。

函数：`fm_refine_sampled`

它的任务是：在不破坏平衡约束的情况下，反复选择 gain 最大的节点移动，并使用 FM 的“最佳前缀回滚”保留最好的移动序列。

### 12.1 基本数据

每一轮 pass 开始时：

1. 重新计算 `count_x/count_y`。
2. 计算当前 `wx/wy`，即两边总权重。
3. 初始化 `locked`，每个节点每轮最多移动一次。
4. 计算所有节点的 gain。
5. 把 `{gain, node, version}` 放入最大堆。

`GainEntry`：

```cpp
struct GainEntry {
    int gain;
    int node;
    int version;
};
```

`version` 用来处理堆里的旧数据。因为 C++ `priority_queue` 不方便原地修改元素，所以代码采用“延迟更新”：

```text
gain 变化 -> version++ -> 推入新条目
弹出堆顶时，如果 version 不是最新版，就丢弃
```

### 12.2 选择移动节点

每一步从最大堆里找 gain 最大且合法的节点。

合法需要满足：

1. 节点没有被锁定。
2. 堆条目版本没有过期。
3. 移动后 `wx` 仍在 `[min_x, max_x]` 范围内。

如果堆里前面若干候选因为平衡约束不合法，会暂存到 `deferred`，稍后放回堆里。

如果检查了限定数量的堆元素仍找不到合法移动，代码会执行一次全量扫描兜底，避免漏掉可行节点。

### 12.3 执行移动与增量更新

选出 `best_u` 后：

1. 标记 `locked[best_u] = 1`。
2. 记录本次移动到 `moves`。
3. 调用 `apply_move` 真正移动节点。
4. 找出受影响邻居。
5. 只重新计算这些邻居的 gain。

受影响邻居是通过两跳找到的：

```text
best_u -> 它连接的 net -> 这些 net 上的其他节点
```

代码用 `dedup_mark` 和 `dedup_stamp` 去重，避免同一个邻居通过多条 net 被重复加入。

### 12.4 最佳前缀回滚

FM 算法允许某些单步 gain 为负，因为连续移动后可能整体更好。

所以代码维护：

```cpp
int cumulative_gain;
int best_prefix_gain;
int best_prefix_len;
```

每移动一步，就把 gain 加到 `cumulative_gain`。

如果当前累计 gain 超过历史最好，就记录当前前缀长度。

一轮结束后：

- 如果 `best_prefix_gain <= 0`，说明整轮没有带来正收益，全部回滚。
- 如果中间某个前缀最好，就把最好前缀之后的移动全部回滚。

这就是 FM 比普通贪心更强的地方：它不是只看单步，而是看一串移动的最佳累计效果。

## 13. 多层 V-cycle 主算法

位置：`Solution::my_partition_algorithm`。

### 13.1 串行算法流程

单次重启的完整流程：

```text
设置随机种子 42 + restart_id * 1000
设置 90 秒 deadline
构造 fine Hypergraph

重复 kRestarts 次：
    levels[0] = fine

    粗化：
        while 层数 < kMaxLevels:
            如果节点数 <= 320，停止
            coarsen_once
            如果粗化无效，停止
            保存 fine_to_coarse 映射
            保存 coarse graph
            如果缩减比例 > 0.92，停止

    在最粗层尝试 3 种初始化：
        初始化 part
        粗层 FM + 粗层 polish

        从粗层逐层反粗化到细层：
            用 fine_to_coarse 映射把粗层 part 投影到更细一层
            中间层用中层 FM 参数细化
            最细层用细层 FM 参数细化

        在原始 fine 图上 evaluate_cut
        如果 cut 更小，保存为当前重启的 best_part

返回该重启找到的最佳 result

在所有重启结果中取全局最优的 part 转回 X/Y
```

其中 `levels[0]` 是原始图，`levels.back()` 是最粗图。

反粗化时的关键代码思想：

```cpp
finer_part[u] = part[map[u]];
```

意思是：细节点 `u` 属于哪个粗节点，就继承那个粗节点的分区。

之后再在更细层继续 FM，逐步修正因为粗化带来的误差。

### 13.2 多线程并行架构

#### 13.2.1 并行化目标的选择

分析整个算法的三层循环结构，从内到外：

```
最内层：FM 单步移动（本质串行，每步依赖上一步的 partition 状态）
         ↓   Amdahl 定律限制：串行部分 = 100%，无法并行
中层：三种初始化策略（共享粗化层次，可并行但规模小）
         ↓  加速比上限 = 3×，需配合外层才能充分利用核心
最外层：kRestarts = 8 次随机重启（完全独立，无任何数据依赖）
         ↓  尴尬并行，理论加速比 = min(kRestarts, 核心数)
```

**结论**：外层重启循环是最理想的并行化目标。

```
为什么外层重启是"尴尬并行"（embarrassingly parallel）？
  每次重启：
  - 从头构建自己的粗化层次（levels 数组线程独占）
  - 使用独立的随机数序列（RNG 线程独占）
  - 运行独立的 FM 搜索路径（part / gain / locked 线程独占）
  - 唯一的共享数据是 fine Hypergraph，且是只读的
  
  → 没有写-写冲突，没有读-写竞争，不需要任何锁来协调运算过程
```

#### 13.2.2 使用 OpenMP 的实现

将串行的循环体提取为独立函数，再通过 `#pragma omp parallel for` 并行化。

**提取为独立函数**（多线程改造的基础重构）：
```cpp
// 一次完整重启，封装为独立函数
// 所有状态都是局部变量（线程独占），不写任何全局变量
RestartResult run_one_restart(int restart_id,
                               const Hypergraph &fine,
                               time_point deadline) {
    mt19937 rng(42 + restart_id * 1000);  // 线程局部 RNG
    // 粗化 → 三种初始化 → FM → 反粗化 ...
    return {best_part, best_cut};
}
```

**OpenMP 并行调用**（替换了原来的分批 + async + mutex 模式）：
```cpp
// 预分配结果数组，每个线程写自己的槽位
vector<RestartResult> results(kRestarts);

#pragma omp parallel for num_threads(num_threads) schedule(dynamic)
for (int r = 0; r < kRestarts; ++r) {
    results[r] = run_one_restart(r, fine, deadline);
}

// 串行合并结果
for (auto &res : results) {
    if (res.cut < best_cut) { best_cut = res.cut; ... }
}
```

与之前 `std::async` 版本的对比：

| 项目 | std::async（之前） | OpenMP（现在） |
|------|-------------------|---------------|
| 并行代码量 | ~40 行 | ~10 行 |
| 线程控制 | 手动分批 + future 同步 | 编译器自动处理 |
| 负载均衡 | 静态分批（尾部等待） | `schedule(dynamic)` 自动均衡 |
| 结果同步 | `std::mutex` 保护 | **无锁**（预分配数组，下标隔离） |
| 运行时依赖 | C++ 标准库 + pthread | 编译器 OpenMP 支持 |

#### 13.2.3 线程安全分析（逐变量）

| 变量 | 作用域 | OpenMP 属性 | 线程安全 | 原因 |
|------|--------|-------------|----------|------|
| `fine` (Hypergraph) | 主线程分配 | **shared**（默认） | ✅ 无竞争 | 构造后永不修改，多线程同时只读安全 |
| `deadline` (time_point) | 主线程计算 | **shared**（默认） | ✅ 无竞争 | 只读 64 位整数，无写操作 |
| `rng` (mt19937) | worker 内创建 | **thread-local** | ✅ 无竞争 | 函数内局部变量，栈上分配，互不干扰 |
| `levels` (vector) | worker 内创建 | **thread-local** | ✅ 无竞争 | 每个 worker 独立构建自己的粗化层次 |
| `part` (vector\<char\>) | worker 内创建 | **thread-local** | ✅ 无竞争 | 分区向量在函数内部，不离开 worker |
| `count_x/count_y` | FM 内创建 | **thread-local** | ✅ 无竞争 | 每次 rebuild 重建 |
| `gain` / `locked` | FM 内创建 | **thread-local** | ✅ 无竞争 | FM 内部状态 |
| `results` | 主线程分配 | **shared** | ✅ **无锁** | 各线程写入 `results[r]` 中不同的 `r` 下标，天然无竞争 |
| `best_part` / `best_cut` | 主线程 | 主线程独占 | ✅ 无竞争 | 仅在串行合并阶段访问 |

**OpenMP 变量属性规则**：
- 循环变量（`r`）自动为 **private**
- 并行区域外声明的变量默认 **shared**
- 函数内声明的局部变量（`run_one_restart` 内部）自动为 **thread-local**
- 因此无需 `std::mutex`，也无需 `#pragma omp critical` 或 `#pragma omp atomic`

#### 13.2.4 OpenMP 调度机制

```text
线程数 = min(num_threads, kRestarts) = 8

#pragma omp parallel for num_threads(4) schedule(dynamic)
for (int r = 0; r < 8; ++r) {
    results[r] = run_one_restart(r, fine, deadline);
}

运行时的调度（假设重启运行时间不同）：

  时间 ──────────────────────────────────────────────►
  
  Thread 0:  restart 0 ────────┐  restart 4 ────────┐
  Thread 1:  restart 1 ───────────┐  restart 5 ─────────┐
  Thread 2:  restart 2 ─────┐     restart 6 ──────┐
  Thread 3:  restart 3 ───────┐   restart 7 ───────┐
                               
  关键：空闲线程自动从任务队列领取下一个未开始的重启
  → 不需要手动分批，没有尾部等待
```

`schedule(dynamic)` 的工作原理：
1. OpenMP 将 `kRestarts` 个迭代放入隐式任务队列
2. 每个线程从队列中取出一个迭代执行
3. 完成后立即取下一个，直到队列为空
4. 天然适应负载不均——没有线程会空闲等待

#### 13.2.5 OpenMP 的变量共享分析

```cpp
#pragma omp parallel for num_threads(num_threads) schedule(dynamic)
for (int r = 0; r < kRestarts; ++r) {
    results[r] = run_one_restart(r, fine, deadline);
}
```

| 变量 | 为什么这样访问 | 安全性 |
|------|--------------|--------|
| `r`（private） | 循环变量，每个线程获得不同的值 | ✅ 安全 |
| `results`（shared） | 各线程写不同下标 `results[r]`，当 `r` 不相同时不会冲突 | ✅ 安全 |
| `fine`（shared） | 只读，所有线程同时读同一个 `Hypergraph` | ✅ 安全 |
| `deadline`（shared） | 只读 64 位整数，编译器可能将其缓存到寄存器 | ✅ 安全 |

#### 13.2.6 随机数种子策略

```cpp
mt19937 rng(42 + restart_id * 1000);
```

- **确定性**：固定初始种子 `42`，每次运行同一 `restart_id` 产生完全相同的随机序列，便于调试和复现问题。
- **独立性**：`restart_id * 1000` 拉开不同重启的随机序列间距（`mt19937` 的状态空间远大于 1000，保证序列不重叠）。
- **多样性**：与串行版本相比，并行版本的随机性不会降低（每个 restart 的 RNG 序列长度不变）。

#### 13.2.7 无锁设计

OpenMP 版本中**不需要任何互斥锁**。原因：

1. 每个线程写入 `results[r]` 中不同的数组下标（`r` 在 OpenMP 中自动为 private）
2. 写入操作是 `RestartResult` 的赋值，不是复合操作
3. 合并阶段在并行区域之后串行执行，不存在竞争

对比之前 `std::async` 版本需要 `std::mutex` 的原因：
- `std::async` 通过 `future.get()` 返回结果，每个线程的写入点无法控制
- 多个线程的 `future` 结果需要在主线程逐一 `get()`，但比较时仍需锁保护共享的 `best_part`/`best_cut`

#### 13.2.8 为什么选择 OpenMP

| 考虑因素 | 决策 |
|----------|------|
| 代码简洁性 | 从 ~40 行降到 ~10 行，编译器自动处理线程生命周期和调度 |
| 性能 | OpenMP 的线程管理开销通常低于 `std::async` + `std::future` 组合 |
| 负载均衡 | `schedule(dynamic)` 自动补偿不同重启的运行时间差异 |
| 可移植性 | g++ / MSVC / ICC / Clang 均支持 OpenMP |
| 与本项目工具链兼容 | MinGW-w64 自带 OpenMP，`-fopenmp` 即可 |

#### 13.2.9 自适应时间预算

当前不再使用固定 90 秒 deadline，而是根据图规模动态计算：

```cpp
// 公式：budget = kBudgetBaseTime × (N / kBudgetBaseNodes)^kBudgetExponent
// 参数在 solution.cpp 顶部定义
int64_t budget_seconds = kBudgetBaseTime;
if (fine.n > kBudgetBaseNodes) {
    double ratio = static_cast<double>(fine.n) / kBudgetBaseNodes;
    budget_seconds = static_cast<int64_t>(kBudgetBaseTime
                                          * pow(ratio, kBudgetExponent));
}
budget_seconds = clamp(budget_seconds, kBudgetMinSeconds, kBudgetMaxSeconds);
auto deadline = steady_clock::now() + seconds(budget_seconds);
```

设计原理：
- **基准**：ibm01（~12k 节点）在 90 秒内可完成 8 次重启 × 3 种初始化的全部搜索
- **缩放指数 0.8**：粗化过程使问题规模每层约减 50-70%，因此有效问题大小的增长是**次线性**的（指数 < 1）
- **上下界**：保证即使极小图也有足够搜索时间（60s），极大图也不会失控（3600s = 1h）

典型值估算：

| 数据集 | 节点数 | 自适应预算 |
|--------|--------|-----------|
| ibm01 | 12k | 90s（基准） |
| ibm05 | 50k | ~287s（5 分钟） |
| case1 | 300k | ~1243s（21 分钟） |
| case8 | 750k | ~2716s（45 分钟） |

超过 budget 后，所有 worker 线程会终止当前重启，返回已找到的局部最优解。因此即使预算不足，程序也始终有输出。

#### 13.2.10 性能模型（Amdahl 定律）

```
加速比 S = 1 / ((1 - P) + P / N)

其中：
  P = 可并行比例（外层重启占总运行时间的比例）
  N = 线程数

对于本算法：
  - 外层重启 ≈ 95% 的运行时间（粗化 + 初始化 + FM + 反粗化）
  - 串行部分 ≈ 5%（fine 构建 + 结果汇总 + X/Y 转换）
  
  N=1:  S = 1.0×
  N=4:  S = 1 / (0.05 + 0.95/4) = 3.5×
  N=8:  S = 1 / (0.05 + 0.95/8) = 5.9×
```

实测数据（ibm01，20 核机器）：

| 线程数 | 实测耗时 | 实测加速比 | Amdahl 理论值 |
|-------|---------|-----------|--------------|
| 1 | 23.5s | 1.00× | 1.00× |
| 2 | 12.9s | 1.82× | 1.90× |
| 4 | 7.7s | 3.05× | 3.48× |
| 8 | 5.3s | 4.43× | 5.90× |

实际加速比略低于理论值，原因：
1. 多线程带来的额外开销（线程调度、缓存失效、内存带宽竞争）
2. 负载不均（不同重启的收敛时间有微小差异，最慢的线程决定批次完成时间）
3. deadline 提前终止（部分线程可能早退，导致有效计算资源浪费）

## 15. 当前实现的特点与注意事项

### 15.1 当前优点

- 已经从简单贪心升级成多层次划分框架。
- 内部 `Hypergraph` 结构更适合高频计算。
- 粗化、初始化、FM 细化模块相对独立，便于继续改进。
- 有多次重启和三种初始划分，能增加搜索多样性。
- FM 使用最大堆、延迟更新、局部 gain 更新，效率比每步全图扫描更高。
- 设置了 90 秒总时间预算，避免大数据集运行时间失控。
- `main.cpp` 已经能自动输出对应 benchmark 的 partition 文件。
- **OpenMP 多线程并行加速**：外层 `kRestarts` 次重启通过 `#pragma omp parallel for schedule(dynamic)` 并行执行，无锁设计（预分配数组做下标隔离），代码仅 ~10 行。8 次重启在 8 核上实测加速 **4-6×**，划分质量不变。

### 15.2 当前局限

- `Graph` 和 `evaluate` 仍然按值传递 `Graph graph`，对大图会有额外复制开销。更合理的写法是 `const Graph &graph`。
- `Graph` 里使用 `new` 创建 `Node/Net`，析构函数没有释放，存在内存泄漏。短程序影响不大，但工程上不够规范。
- `Graph::get_or_create_node` 已修复为 `node_map`（O(log n)）查找。
- `main.cpp` 生成输出文件名现在同时识别 `/` 和 `\` 路径分隔符。
- 时间预算已改为**自适应**，根据图规模动态计算（公式：`90 × (N/12000)^0.8`），不再使用固定参数。
- 多线程方面：三种初始化暂未并行（在重启并行已充分利用核心的场景下意义不大）；FM 内部的 `build_net_side_counts` 和初始 gain 计算未做 SIMD 向量化（受益有限）。

struct WorkItem { int restart_id; int init_mode; };
vector<WorkItem> items;
for (int r = 0; r < kRestarts; ++r)

## 16. 一句话总结

当前项目已经形成了比较完整的划分器：

```text
读取 ICCAD 2021 case 文件
-> 构造 Graph（0 索引 -> 1 索引转换）
-> 转成内部 Hypergraph
-> 多线程并行执行 kRestarts 次重启：
    每线程独立完成：多层粗化 -> 最粗层三种初始化 -> FM 细化 -> 逐层反粗化
-> 汇总所有线程结果，取 cutsize 最小解
-> 输出 results/caseX_partition.txt
-> evaluate 计算 cutsize
```

它的核心思想是：先在粗图上快速找到一个大致合理的全局划分，再逐层回到原图，用 FM 细化不断修正局部决策，从而在满足平衡约束的前提下降低割代价。
